# YouTube Shorts 自動編集・投稿プロジェクト（Marbles Arena）

## プロジェクト概要
- チャンネル：Marbles Arena【自作ルーレットゲーム】
- 内容：自作のマーブルラン（ルーレット）で、どの色の球が入るかを視聴者に予想させるゲーム動画
- 元素材の形式：長尺の録画MP4（複数の色の球が絡むレース。ファイル名に色ラベルが付く。例：N021_ピンク黃緑オレンジ_ピンクオレンジ_緑場外.mp4）
- 完成品の実例：https://youtube.com/shorts/aI0ZuVigtiE
  - タイトル：「3秒で入る球予想してね！（正解は最後！) s2p11 #shorts」
  - 長さ：**20秒**（Shorts上限の60秒よりかなり短い。予想→結果発表がテンポよく収まる長さを優先する）
  - カテゴリ：Gaming
- 投稿頻度：週2〜3本

## 動画仕様

### 入力素材（input/）の実測値
- 解像度：1080×1920px（縦型 9:16）
- フレームレート：60fps
- 映像コーデック：H.264
- 音声：AAC 48kHz ステレオ
- 長さ：約60〜61秒（長尺の録画。この中から「予想が的中するか」の見どころ部分だけを切り出して使う）
- ビットレート目安：6Mbps前後
- ファイル名に色ラベル（ピンク／黄緑／オレンジ／緑など）が付いており、レースに絡む球の色を表す

### 出力（output/）の統一仕様（実例 s2p11 に準拠）
- 解像度：1080×1920px（変更しない。入力が異なる比率の場合はクロップ＋パディングで合わせる）
- フレームレート：30fps に統一して出力（60fps素材は30fpsへ変換）
- **長さ：固定ではなく可変（目安20〜30秒程度）**。実例でも s2p11=20秒、s2p8=28秒と幅がある。球が動き出してから結果が出るまでの実際の時間に合わせる（60秒の上限にこだわらない）
- **構成（3段階、この順序・切り替えを厳守）**：
  1. **煽りフェーズ**：「3秒で入る球予想してね！」等の煽りテキストを表示（球が動き出す前の静止〜導入部分に重ねる）
  2. **レースフェーズ**：球が動き出し、コースを転がっていく本編。ここは編集でカットしすぎず、球の動きが自然に見える尺を確保する
  3. **結果フェーズ**：どの色の球が入ったか結果が分かるカットで締める。「正解は最後！」に矛盾しないよう、結果が明確に映る瞬間で終わる（必要なら結果を強調するテキストを追加）
- 音声：AAC 44.1kHz、BGM（緊張感を煽る系）を原音より小さめに乗せる（volume=0.3を基準値とする）
- フォーマット：MP4（H.264 + AAC、faststart指定でアップロード後の再生をスムーズに）

## 見どころ区間の切り出しルール（半自動フロー）
長尺素材（約60〜61秒）から「煽り→レース→結果」の3段構成に使う区間を切り出す作業は、以下の半自動フローで行う。**尺は固定20秒ではなく可変**（球が動き出してから結果が出るまでの実時間に合わせ、目安20〜30秒程度）。

1. **仮切り出しのルール**：
   - 素材内で球が動き出す直前の位置（＝煽りフェーズの開始点）を仮検出する。初期ルールとしては**末尾から30秒**の位置を候補開始点とする（例：全体が61秒なら31秒目あたりから）
   - 候補開始点〜末尾までを仮切り出しし、プレビューとして提示する
2. **確認**：仮切り出しした候補区間（またはプレビュー用の静止画数枚）を提示し、以下を確認してもらう
   - 球が動き出す瞬間から結果までが過不足なく含まれているか（間延びがないか／逆に途中から始まっていないか）
   - 結果が決まる瞬間がちゃんと含まれているか
3. **調整**：ズレがあれば開始位置を手動で前後に調整し、再度候補を提示する
4. **確定後**：確定した区間をもとに `skills/edit_video/SKILL.md` の本編集（煽りテキスト焼き込み・BGM合成・結果テキスト）に進む

※ 「末尾から30秒」はあくまで初期ルール。実際の尺は20〜30秒程度でばらつくため、運用しながら候補開始点の算出方法（比率にする、球が動き出すモーション検出にする等）を見直す

## 使用ツール
- FFmpeg（動画編集）
- Python 3.x（自動化・スクリプト実行）
- YouTube Data API v3（アップロード、google-api-python-client / google-auth-oauthlib を使用）

## 禁止事項・注意事項
- 著作権のある音楽・映像は使用しない（BGMは著作権フリー素材のみ、bgm/フォルダで管理）
- **BGMは新規選定不要**：`bgm/` フォルダに入っている既存の音源をそのまま使い続ける（前回動画から引き続き使用）。新しい音源を追加する場合のみユーザーに確認する
- APIキー・クライアントシークレットはコードに直書きせず `.env` で管理する（.envは.gitignoreに追加）
- output/ フォルダのファイルは上書きしない。ファイル名は `YYYYMMDD_HHMMSS_連番.mp4` 形式でタイムスタンプ付き保存
- 同じ動画を二重投稿しない（uploaded_log.csv で投稿済みIDを必ず確認してからアップロード）
- 入力素材のファイル名にある色ラベル（ピンク／黄緑／オレンジ／緑など）は、レースに絡む球の色として扱い、必要に応じて説明文・タグ付けの参考情報にする（タイトルへの直接反映は不要。タイトルは自動採番ルールに従う）
- `uploaded_log.csv` には投稿日時・元ファイル名・動画IDに加えて、採番した `season` / `part` を必ず記録する（次回採番のため）

## タイトルの自動採番ルール
- 形式：「3秒で入る球予想してね！（正解は最後！) s{season}p{part} #shorts」
- **シーズン（season）は固定運用**：明確な繰り上げルールは現状ないため、season番号は変更せず `part` のみ加算し続ける
  - 前回投稿から間が空いたため、**次回投稿から `season=3` に切り替える**（今回のみ手動対応）
  - 次回投稿は `s3p1` から開始し、以降は `s3p2`、`s3p3` ... と同シーズン内で連番を続ける（当面 `season` は3で固定）
  - 将来的にシーズンを繰り上げる場合は、その都度ユーザーに確認してから手動で切り替える（自動繰り上げロジックは実装しない）
- 採番ロジックは `scripts/upload.py` 側で実装し、投稿ごとに `uploaded_log.csv` へ記録した最新の `part` を読み取って次の番号を決定する（`season` は固定値として扱う）

## Git運用ルール（Claudeへの指示）
- 新規機能の実装や修正が発生した場合は、master で直接作業せず、新しくブランチを作成してそちらで作業する
  - ブランチ名：`feature/実装内容が分かる名前`（例：`feature/edit-upload-scripts`、`feature/fix-bgm-loop`）
- 実装が完了したら、ブランチを push し、master への Pull Request を作成するところまで行う
- PR のマージはユーザーが内容を確認してから行う（Claude は自動でマージしない）

## 作業の進め方（Claudeへの指示）
1. 新しい編集・投稿タスクを頼まれたら、まず `skills/edit_video/SKILL.md` を読んでから作業する
2. 見どころ区間は「見どころ区間の切り出しルール」に従い自動で仮切り出しし、**本編集に進む前に必ずユーザーへ確認を取る**
3. 投稿作業を頼まれたら、まず `skills/upload_youtube/SKILL.md` を読んでから作業する。**説明文1行目の一言コメントは動画ごとに手動で変えるため、投稿ごとにユーザーへ確認する**
4. シーズン（season）は固定運用（当面`season=3`）。次回投稿は`s3p1`から開始し、以降は`part`のみ加算する。将来シーズンを切り替える場合は必ずユーザーに確認する
5. 仕様に反する変更（解像度変更・尺の大幅な超過など）が必要な場合は、実行前に必ず確認を取る
